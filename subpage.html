Problem Statement & Vision

The Challenge: An Information Labyrinth in Corporate Folders

At a larger enterprise (in this case, Szeged Waterworks), finding internal policies, integrated management manuals, and documents often feels like wandering through a maze for new or less experienced colleagues. Searching for the right information consumed a significant amount of valuable working time.

The Primary Goal and the Necessity of Local Execution

My original idea was to create an intelligent search system (RAG - Retrieval-Augmented Generation) that tells you which document to look for regarding a specific topic. Since we are dealing with strictly confidential, internal corporate data, cloud-based solutions (e.g., OpenAI API) were completely out of the question. Therefore, I designed the project from the ground up to run 100% locally, using open-source models (Ollama), guaranteeing maximum data privacy and security.

From MVP to a Real Assistant

Once the initial MVP (Minimum Viable Product) was ready, the next step became obvious: why settle for the model merely pointing out the document's location? Let's develop it further so it directly answers the user's question based on the source materials! This, however, required a robust Data Ingestion pipeline capable of handling a wide variety of formats: scanned PDFs, Word documents, and Excel spreadsheets.

Data Ingestion

The Importance of Quality Data

A RAG (Retrieval-Augmented Generation) system is only as good as the quality of the data fed into it. The "Garbage in, garbage out" principle heavily applies here: if the model receives poorly structured, noisy text, the answers will be inaccurate or useless. The biggest challenge in the corporate environment was dealing with scanned PDFs and mixed-format (Word, Excel) documents.

Structured Extraction using Marker

To extract raw text, I deployed a real "heavyweight," the open-source Marker library. Unlike traditional OCR solutions, Marker doesn't just recognize characters from scanned PDFs; it generates a structured Markdown format. It can preserve headers, paragraphs, and lists, which is essential for maintaining context during the subsequent chunking phase. Additionally, I prepared the system to handle .docx and .xlsx files, automatically converting them to PDF in the background before Marker processes them.

The "Extra Mile": LLM-based Proofreading

Scanned documents (OCR) often contain misreads and character errors. To eliminate these, I built an optional step into the pipeline: I used a dedicated Ollama model (Llama 3.1) as a "proofreader". Using a strictly written System Prompt, the LLM corrects grammatical and spelling errors in the extracted Markdown text, while being strictly forbidden from changing the sentence structure, formatting, or the actual data. Thus, only fully cleaned, high-quality text enters the vector database.

Chunking and ChromaDB

Markdown-based Structuring and Experimentation

After extracting quality data, the next critical step was the proper segmentation (chunking) of the text. Since Marker already provided structured Markdown, the solution was straightforward: I divided the documents into logical units based on Markdown headers (H1, H2, H3). After extensive experimentation and studying industry RAG best practices, I set the optimal chunk sizes and overlaps so that the LLM always receives the most relevant and complete context without losing the thread.

Version Control and Automatic Updates (Production-ready feature)

At a large company, policies are constantly updated. I knew that if the system went live, manually maintaining the database would be impossible. Therefore, I wrote a custom, intelligent filename parser logic for the vector database (ChromaDB). Upon execution, the system recognizes document versions (based on date stamps):

  • If it finds a newer version, it automatically locates and deletes the outdated chunks belonging to the old document from the database.
  • Only then does it perform the chunking, embedding (using the bge-m3 model), and uploading of the new version. This ensures the assistant never hallucinates outdated information.

Rich Metadata and Overcoming "Dependency Hell"

To make the system searchable, I saved rich metadata alongside the chunks during the upload process (e.g., source file name, file path). The technical implementation of this wasn't smooth sailing: coordinating Marker, LangChain, Ollama, and various PDF processing libraries quickly turned into a "dependency hell". To avoid conflicts and ensure portability, I organized the different data processing pipelines into dedicated, isolated Python virtual environments (venv).

UI, Prompts, and Metrics

Clean UI and Transparency (Streamlit)

For a prototype, command-line (CLI) execution is sufficient, but end-users needed an easy-to-use, clean interface. My choice was Streamlit, which allowed me to quickly and effectively provide a professional web interface for the assistant. To build user trust in the system, the UI not only displays the final answer but also shows the raw chunks retrieved from ChromaDB (which serve as the basis for the answer) and their metadata in an expandable format.

Strict Prompt Engineering (Eliminating Hallucinations)

The biggest danger of RAG systems is that the model "invents" (hallucinates) information not present in the source. To prevent this, I established an extremely strict set of rules (Prompt). The LLM receives the asked question and the retrieved chunks, and the prompt forces it: it must work exclusively from the provided information. If the context does not contain the answer, the model is obligated to state this plainly, rather than attempting to guess.

Feedback Loop and SQL Telemetry

For continuous system improvement and evaluation, I also implemented a built-in measurement system:

  • Execution time monitoring: The interface separately measures and displays how long the database search took (Retrieval time) and how much time the model took to generate the answer (Generation time).
  • User evaluation: A Like/Dislike button is located under every answer to gather feedback.
  • Database logging: The system structurally saves these metrics (the asked question, the generated answer, the number of chunks used, execution times, and user evaluation) into an SQL database. This database is essential for future fine-tuning of the system and filtering out potential search anomalies.

Summary and Scalability (MVP Status)

The Current Phase: A Solidly Built MVP

It is important to emphasize that the presented project in its current form is an MVP (Minimum Viable Product). Although the architecture and the entire end-to-end process (from data ingestion through vectorization to generation) work flawlessly, the longer model execution and response times visible in the attached images and metrics stem from the current environment.

The Hardware Cost of Local Execution

Since data privacy was the top priority during the system's design, the entire process—including running the Llama 3.1 model with the Ollama framework—happened locally on my own development machine. Running such a complex, 8-billion parameter large language model (LLM) and the embedding model simultaneously on a personal computer naturally requires more computational capacity and time.

Future Development and the Path to Production

This current "bottleneck" can be easily resolved during a production deployment. The code and the RAG pipeline are already ready to scale: moving the system to dedicated on-premise AI servers (or a private cloud) equipped with appropriate GPUs would reduce the inference (answer generation) time to a fraction. The main goal of the project was the "Proof of Concept" and creating a secure, internal data management logic—which the system fully achieves.
